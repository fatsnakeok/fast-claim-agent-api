# 智能保险平台 — 业务分析

> 通过与ai对话，梳理业务流程和开发计划。

## 1. 业务域概述

本项目是一个**车险智能服务平台**，覆盖车险投保、核保、理赔、客服四大核心业务场景。系统通过 Embabel Agent 框架驱动 LLM（DeepSeek）实现智能核保决策、欺诈检测、Agentic RAG 客服问答。

### 1.1 角色体系

| 角色 | 职责 | 系统用户 |
|------|------|---------|
| **普通用户** | 提交投保申请、发起理赔、咨询客服 | user |
| **核保员** | 审核中风险投保申请、审批报价单 | underwriter |
| **理赔员** | 审核存疑理赔单、批准或拒绝赔付 | claims |
| **管理员** | 管理知识库文档、全权限 | admin |

## 2. 核心业务流程

### 2.1 核保流程（Underwriting）

```
用户提交投保申请（自然语言描述车型 + 投保需求）
  │
  ├─ [1] LLM 提取车辆信息（品牌/型号/车牌）
  │      ├─ Guardrail 校验：输入是否包含车辆关键词
  │      └─ 失败 → ERROR (ExtractionFailed)
  │
  ├─ [2] 查找客户档案（按 userId 查 DB）
  │      └─ 未找到 → ERROR (CustomerNotFound)
  │
  ├─ [3] 查找车辆信息（车牌 → 车型 → 客户唯一车辆）
  │      └─ 未找到 / 多车歧义 → ERROR (VehicleLookupError)
  │
  ├─ [4] 风险评估（6因子评分：年龄 + 驾龄 + 事故 + 车龄 + 车价）
  │      │
  │      ├─ riskScore ≤ 60  → ✅ 自动批准 → 生成报价单 (APPROVED)
  │      ├─ 60 < score < 80 → ⏳ 转人工审核 → 生成报价单 (REFERRED)
  │      └─ score ≥ 80      → ❌ 自动拒绝 → 生成报价单 (DECLINED)
  │
  └─ 输出：UnderwritingResult (quoteId, status, riskScore, premium, message)
```

**业务规则**：
- 报价单有效期 30 天
- 保费 = 车价 × 2% × 风险系数 × 险种系数
- REFERRED 报价单需核保员审批后才能支付

### 2.2 支付签单流程（Payment & Policy Issuance）

```
用户对 APPROVED 报价单发起支付
  │
  ├─ 校验：报价单状态 = APPROVED + 未过期
  ├─ 模拟支付处理
  ├─ 生成保单号：POL-{timestamp}-{random}
  ├─ 创建保单（默认一年有效期，状态 = ACTIVE）
  │
  └─ 输出：Policy (policyNumber, premiumAmount, effectiveDate, expirationDate)
```

**业务规则**：
- 仅 APPROVED 状态可支付
- 过期报价单不可支付
- 保单有效期默认一年

### 2.3 人工审批流程（Manual Quote Approval）

```
核保员对 REFERRED 报价单进行审批
  │
  ├─ 校验：报价单状态 = REFERRED + 未过期
  ├─ 可覆盖保费金额（可选）
  ├─ 状态变更：REFERRED → APPROVED
  │
  └─ 输出：ApproveQuoteResponse
```

### 2.4 理赔流程（Claims）

```
用户提交理赔申请（保单号 + 事故描述 + 理赔金额）
  │
  ├─ [1] 保单校验（存在 + ACTIVE + 有效期内）
  │      └─ 失败 → ERROR (PolicyError)
  │
  ├─ [2] LLM 提取事故信息（类型/地点/日期/涉及方）
  │      ├─ LLM 提取成功 → ClaimInfo
  │      └─ LLM 失败 → 关键词回退提取
  │
  ├─ [3] 欺诈评分计算（4因子）
  │      ├─ 理赔/保费比 + 绝对金额 + 信息完整度 + 事故类型
  │      └─ 评分钳制在 [0, 100]
  │
  ├─ [4] 重复检测：同保单 + 同描述 → ERROR (DuplicateClaimDetected)
  │
  ├─ [5] 分类路由：
  │      ├─ fraudScore < 30  → ✅ AutoApproved（自动批准，立即赔付）
  │      ├─ 30 ≤ score < 70  → ⏳ PendingReview（转人工审核）
  │      └─ fraudScore ≥ 70  → ❌ AutoDenied（自动拒绝）
  │
  └─ 输出：ClaimResult (claimNumber, status, fraudScore, approvedAmount, message)
```

**业务规则**：
- 赔付上限 = 保费 × 5 倍（限额封顶）
- 理赔金额低于上限则全额赔付
- 重复理赔（同保单+同描述）自动拒绝
- INVESTIGATING 状态理赔需理赔员通过独立审核接口处理（批准/拒绝/调整金额）

### 2.5 理赔审核流程（Manual Claim Review）

```
理赔员对 INVESTIGATING 理赔单进行审核
  │
  ├─ 校验：理赔状态 = INVESTIGATING
  ├─ 决定：APPROVED 或 DENIED
  ├─ APPROVED → 设置赔付金额（可覆盖但不超过上限）
  ├─ DENIED → 赔付金额 = 0
  │
  └─ 输出：ReviewClaimResponse
```

### 2.6 AI 客服流程（Chatbot / Agentic RAG）

```
用户发送保险咨询问题
  │
  ├─ 会话管理：自动创建/续期（TTL 30min，最多保留 20 轮对话）
  ├─ 输入护栏校验：注入检测、无关话题过滤
  ├─ Agentic RAG 问答：
  │   ├─ LLM 自主调用 insurance_docs_textSearch 工具检索知识库
  │   ├─ BM25 全文检索（保险合同、理赔指南、FAQ）
  │   ├─ LLM 综合检索结果生成回答
  │   └─ 回复护栏校验：敏感信息、幻觉检测
  │
  └─ 输出：ChatResponse (response, sessionId, isNewSession)
```

**知识库文档**：
- `comprehensive_vehicle_insurance.md` — 综合车辆保险条款
- `claims_guide.md` — 理赔指南
- `faq.md` — 常见问题

## 3. 业务实体模型

```
Customer (1) ──── (N) Vehicle       客户拥有多辆车
Customer (1) ──── (N) Policy        客户持有多份保单
Customer (1) ──── (N) Quote         客户有多个报价单
Policy    (1) ──── (N) Claim        保单对应多次理赔
Vehicle   (1) ──── (N) Policy       车辆关联多份保单
Vehicle   (1) ──── (N) Quote        车辆关联多个报价单
```

### 3.1 实体状态机

**Quote 状态流转**：
```
PENDING → APPROVED ──→ (支付) ──→ Policy
        → REFERRED ──→ (人工审批) ──→ APPROVED
        → DECLINED (终态)
```

**Claim 状态流转**：
```
PENDING → INVESTIGATING ──→ (自动/人工批准) ──→ APPROVED ──→ PAID
                          → (自动/人工拒绝) ──→ DENIED (终态)
```

**Policy 状态流转**：
```
ACTIVE → EXPIRED / CANCELLED / SUSPENDED
```

## 4. 核心业务规则汇总

### 4.1 核保风险评分（RiskCalculationService）

| 因子 | 条件 | 加减分 |
|------|------|--------|
| 年龄 | < 25 | +25 |
| 年龄 | 25–35 | +15 |
| 年龄 | 36–64 | 0 |
| 年龄 | ≥ 65 | +20 |
| 驾龄 | < 3 年 | +20 |
| 驾龄 | 3–5 年 | +10 |
| 驾龄 | 6–19 年 | 0 |
| 驾龄 | ≥ 20 年 | −10 |
| 事故次数 | 每次 | +15 |
| 车龄 | > 10 年 | +15 |
| 车龄 | 6–10 年 | +8 |
| 车辆价值 | > ¥500,000 | +10 |

**阈值**：≤ 60 批准 · 61–79 转人工 · ≥ 80 拒绝

### 4.2 保费计算（PremiumCalculationService）

```
保费 = 车辆价值 × 2% × 风险系数 × 险种系数

风险系数：低风险(< 40) ×0.8 | 中风险(40-69) ×1.0 | 高风险(≥ 70) ×1.5
险种系数：第三者责任险 ×0.5 | 盗抢险 ×0.75 | 综合险 ×1.0
```

### 4.3 欺诈评分（ClaimsAgent）

| 因子 | 条件 | 加分 |
|------|------|------|
| 理赔/保费比 | > 10× | +40 |
| 理赔/保费比 | > 5× | +20 |
| 理赔/保费比 | > 3× | +10 |
| 绝对金额 | > ¥100,000 | +30 |
| 绝对金额 | > ¥50,000 | +15 |
| 涉及方不明 | "unknown" | +20 |
| 日期缺失 | 空 | +15 |
| 地点缺失 | 空/unknown | +10 |
| 事故类型 | theft | +10 |

**阈值**：< 30 自动批准 · 30–69 转人工 · ≥ 70 自动拒绝
**赔付上限**：保费 × 5

## 5. 护栏体系

| 护栏 | 层级 | 校验内容 |
|------|------|---------|
| InsuranceUserInputGuardRailImpl | 输入 | 注入指令（skip verification, bypass review 等）、提示注入（Base64 编码指令）、无关话题（stock, weather, politics 等） |
| VehicleInfoGuardRail | 输入 | 车辆信息关键词校验（品牌+型号或车牌） |
| InsuranceAssistantMessageGuardRailImpl | 输出 | 敏感信息泄露（信用卡号、SSN）、幻觉迹象（不确定表述）、回复长度异常（< 10 或 > 5000 字符） |

## 7. 开发计划

### 7.1 已完成 ✓（按开发顺序）

#### 阶段一：数据层与基础设施

| 模块 | 功能 |
|------|------|
| **数据初始化** | 实体建模（Customer/Vehicle/Policy/Quote/Claim/PolicyDocument）+ JPA Repository + 3种风险等级种子数据 |
| **安全认证** | HTTP Basic Auth + 4角色层级（ADMIN > UNDERWRITER/CLAIMS > USER）+ 方法级 @PreAuthorize |
| **LLM 分层** | fast/balanced/powerful 三档模型角色路由，对应 DeepSeek Chat/Reasoner |

#### 阶段二：领域服务（纯业务逻辑，无框架依赖）

| 模块 | 功能 |
|------|------|
| **风险评分服务** | 6因子评分算法（年龄/驾龄/事故/车龄/车价），钳制 [0, 100] |
| **保费计算服务** | 保费 = 车价 × 2% × 风险系数 × 险种系数 |
| **数据查询服务** | 客户/车辆多维度查询，精确匹配 + 模糊回退 |
| **LLM 选择服务** | 按任务类型/复杂度自动路由模型 |

#### 阶段三：核心 Agent（Embabel 框架接入）

| 模块 | 功能 |
|------|------|
| **核保Agent** | LLM 提取车辆信息 → 客户/车辆查找 → 风险评估 → @State 三级路由（批准/转人工/拒绝）+ 3条错误路由 |
| **理赔Agent** | 保单校验 → LLM 提取事故信息 → 4因子欺诈评分 → 重复检测 → @State 路由（自动批准/转人工/自动拒绝）+ 3条错误路由 |
| **客服Agent** | Agentic RAG 单动作（ToolishRag 注册 Lucene 搜索工具，LLM 自主检索→综合→回答） |

#### 阶段四：业务编排与 API 暴露

| 模块 | 功能 |
|------|------|
| **Agent 编排服务** | AgentProcess 生命周期管理、120s 超时保护、StuckHandler 诊断、Blackboard 错误回退 |
| **支付服务** | 报价单状态校验 → 模拟支付 → 保单签发（POL-{timestamp}-{random}，一年有效期） |
| **人工审批** | REFERRED 报价单审批流程，支持核保员覆盖保费 |
| **理赔审核** | INVESTIGATING 理赔单批准/拒绝，支持覆盖赔付金额（不超过上限） |
| **会话管理** | 多轮对话会话生命周期（TTL 30min），历史截断（20轮），AgentInvocation 调用 |
| **Controller 层** | REST API：核保/理赔/支付/审批/审核/保单查询/客服对话/RAG 管理，共 12 个端点 |

#### 阶段五：知识库与 RAG

| 模块 | 功能 |
|------|------|
| **RAG 检索引擎** | LuceneSearchOperations BM25 全文检索（纯文本，chunk=1000，overlap=200） |
| **文档摄入服务** | Markdown → MaterializedDocument 层级解析 → Lucene 分块索引，支持全部/增量/单文档摄入 |
| **RAG 管理接口** | 摄入状态查询、触发摄入、重新摄入、可用文档列表（仅 admin） |
| **启动自动摄入** | DocumentIngestionRunner 启动时自动将 3 份知识库文档写入 Lucene 索引 |

#### 阶段六：横切关注点

| 模块 | 功能 |
|------|------|
| **护栏体系** | 三层防护：输入注入检测 + 车辆信息 GuardRail + 输出幻觉/敏感信息检测 |
| **缓存系统** | 二级缓存（Spring Cache llm-responses/rag-searches + 本地 ConcurrentHashMap TTL 5min），定时清理 |

#### 阶段七：测试
